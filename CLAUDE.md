## 세션 시작 시 필독 파일

새 세션을 시작할 때 **아래 파일을 순서대로 반드시 모두 읽는다.** 한 파일만 읽고 넘어가지 말 것. 각 파일은 역할이 다르며 서로를 대체하지 않는다.

| 순서 | 파일 | 역할 | 필독 이유 |
|:---:|---|---|---|
| 1 | `CLAUDE.md` (본 파일) | 코딩 규칙·소통 원칙·기술 스택·로드맵 체크리스트 | 작업 방식과 제약을 먼저 파악 |
| 2 | `PROJECT_PLAN_v2.md` | 기능 명세·데이터 모델·UI 흐름·상세 로드맵 | 무엇을 만들지, 왜 그렇게 만드는지 파악 |

**우선순위 규칙**: 두 문서 사이 내용이 상충하면 `CLAUDE.md` (본 파일)이 우선한다. `PROJECT_PLAN_v2.md`는 기능·제품 관점의 진술이고, `CLAUDE.md`는 구현·운영 관점의 규칙이므로 후자가 최종 결정권을 가진다.

**갱신 규칙**: 로드맵 체크박스, 작업 로그, 신규 결정사항은 `CLAUDE.md`에 반영한다. 기획 변경(새 기능, 데이터 모델 변경 등)은 `PROJECT_PLAN_v2.md`에도 병행 반영한다.

## 프로젝트 개요

**이름**: vape-crm (전자담배샵 CRM)

**목적**: 기존 엑셀(.xlsb) 기반 CRM을 Go + SQLite 기반의 **단일 실행 웹 애플리케이션**으로 재구축.

**상세 기획서**: `PROJECT_PLAN_v2.md` — 전체 기능/로드맵은 이 파일 참조.

**개발자**: 주니어 (1~3년차). 학습과 완성을 동시에 추구.

## 사용자와의 소통 원칙

**중요**: 사용자는 Go 주니어 개발자이다. 아래 원칙을 반드시 지킬 것.

1. **답변은 한국어로** 한다. 코드와 명령어는 영문 그대로.
2. **코드를 바로 쓰지 말 것.** 먼저 접근 방법을 설명하고, 사용자가 이해했는지 확인한 뒤 구현.
3. **새 개념이 등장하면 짧게 설명**을 덧붙인다 (예: "트랜잭션이란..."). 다만 지나치게 장황하지 않게.
4. **파일을 수정한 경우 변경 내역을 요약**해서 사용자에게 설명한다.
5. **Plan Mode를 적극 활용**한다. 복잡한 변경 전에는 먼저 계획을 보여주고 승인받기.

## 기술 스택 (확정됨, 변경 금지)

| 계층 | 기술 | 비고 |
|---|---|---|
| 언어 | Go 1.22+ | 단일 바이너리 빌드 |
| 웹 라우터 | `github.com/go-chi/chi/v5` | 표준 net/http 호환, 경량 |
| DB | **SQLite** | 외부 DB 금지 |
| DB 드라이버 | `modernc.org/sqlite` | **CGO 사용 금지** → 크로스 컴파일 유지 |
| 템플릿 | Go 표준 `html/template` | SSR |
| 프론트엔드 | **HTMX + Alpine.js** | SPA 빌드 도구 사용 금지 |
| CSS | Pico.css (CDN) | 빌드 파이프라인 없음 |
| 차트 | Chart.js (CDN) | 결산 페이지용 |

### 사용 금지 목록

- **유료 SaaS / 클라우드 DB** (PostgreSQL, MySQL 등 별도 설치 필요 DB 포함)
- **CGO 의존 라이브러리** (예: `github.com/mattn/go-sqlite3`) — 크로스 컴파일 어려워짐
- **React, Vue, Svelte 등 SPA 프레임워크** — HTMX로 충분
- **npm, webpack 등 Node 빌드 도구** — Go 바이너리 하나로 배포 원칙 유지
- **ORM** (GORM 등) — 학습 목적 + SQL 직접 작성이 LIKE 검색 등 세밀한 제어에 유리

## 프로젝트 구조

```
vape-crm/
├── cmd/server/main.go            # 진입점
├── internal/
│   ├── config/                   # 환경 설정
│   ├── db/
│   │   ├── db.go                 # SQLite 연결, 초기화
│   │   └── migrations/           # 001_init.sql 등
│   ├── models/                   # 구조체 정의
│   ├── repository/               # DB 접근 계층
│   ├── service/                  # 비즈니스 로직 (트랜잭션)
│   ├── handler/                  # HTTP 핸들러
│   └── search/                   # 부분일치·초성 검색 공통 모듈
├── web/
│   ├── templates/                # HTML 템플릿
│   └── static/                   # CSS/JS/이미지
├── data/                         # SQLite 파일 저장 위치 (.gitignore)
├── PROJECT_PLAN_v2.md            # 기능 기획서
├── CLAUDE.md                     # 본 파일
└── go.mod
```

**디렉토리 규칙**
- `internal/`은 외부에서 import 못 하게 하는 Go 표준 관행 — 모든 애플리케이션 코드는 여기 둔다.
- 비즈니스 로직은 **반드시 `service/`에** 둔다. `handler/`에서 DB를 직접 만지지 말 것.

## 코딩 원칙

### 일반

- **함수는 짧게** — 원칙적으로 30줄 이내. 넘어가면 분리.
- **의미 있는 변수명** — `tmp`, `data` 같은 이름 금지. `customerID`, `monthlySales` 처럼.
- **주석은 "왜"를 설명** — "무엇을 하는지"는 코드로 보이게.
- **에러는 반드시 처리** — `_ = err` 같은 무시 금지. 최소한 로깅.

### 금액 처리 (중요)

- **모든 금액은 `int64` (원 단위 정수)** 로 저장 및 계산.
- `float64` 사용 금지 — 원화는 소수점이 없고, 부동소수점은 오차가 발생.
- 예: `PurchasePrice int64` (3190원), `SaleAmount int64` (31000원)

### SQL

- **반드시 prepared statement 사용** — SQL 인젝션 방지.
  ```go
  // ✅ 올바름
  db.Query("SELECT * FROM products WHERE name LIKE ?", "%"+keyword+"%")

  // ❌ 금지
  db.Query("SELECT * FROM products WHERE name LIKE '%" + keyword + "%'")
  ```
- **매출/매입은 반드시 트랜잭션** 안에서 재고 변동까지 함께 처리.
- **복잡한 쿼리는 SQL 파일로 분리**하거나 최소 상수로 정의하여 재사용성 확보.

### Go 관용구

- 에러는 마지막 반환값으로: `func Foo() (Result, error)`
- 리소스는 `defer`로 정리: `defer rows.Close()`
- 컨텍스트는 첫 번째 인자: `func (r *Repo) Find(ctx context.Context, id int64)`
- 인터페이스는 사용하는 쪽에서 정의 (의존성 주입)

## 핵심 기능: 부분일치 검색 (최우선)

이 프로젝트의 **킬러 피처**이자 기존 엑셀의 가장 큰 불편함을 해결하는 기능.

### 구현 원칙

1. **모든 검색 필드에 동일한 UX** 적용 — 품목, 고객 모두.
2. **HTMX 실시간 검색** — 타이핑 시 200ms 지연 후 서버 요청.
3. **SQL LIKE + 정렬 우선순위**:
   - 1순위: 앞쪽 일치 (`name LIKE '검색어%'`)
   - 2순위: 포함 일치 (`name LIKE '%검색어%'`)
4. **재사용 가능한 모듈**로 `internal/search/`에 구현.

### 표준 검색 쿼리 템플릿

```sql
SELECT id, name, ...
FROM {table}
WHERE name LIKE '%' || ? || '%'
ORDER BY
    CASE WHEN name LIKE ? || '%' THEN 1 ELSE 2 END,
    name
LIMIT 20;
```

### 2차 개선: 초성 검색

- 품목 등록/수정 시 초성 추출해 `name_chosung` 컬럼에 저장.
- 검색어가 초성만 포함하면 `name_chosung` 컬럼으로 검색.
- 구현은 MVP 완료 후 진행.

## Git 커밋 컨벤션

### 메시지 형식

```
[타입] 짧은 요약 (50자 이내)

(선택) 본문: 왜 이렇게 바꿨는지
```

### 타입

- `feat`: 새 기능
- `fix`: 버그 수정
- `refactor`: 리팩토링 (기능 변화 없음)
- `docs`: 문서 변경
- `test`: 테스트 추가/수정
- `chore`: 빌드, 설정 등 기타

### 예시

- ✅ `feat: 품목 부분일치 검색 구현`
- ✅ `fix: 매출 등록 시 재고가 차감되지 않는 문제 수정`
- ✅ `refactor: 매출 서비스의 트랜잭션 처리 분리`
- ❌ `수정함`, `test`, `update`

### 커밋 시점

- **작동하는 상태**마다 커밋 — 실패한 상태로 커밋 금지.
- **의미 있는 단위**로 커밋 — "오타 수정" + "새 기능" 같이 묶지 말 것.
- 사용자에게 "이제 커밋하는 게 좋겠다"고 제안할 것.
